# Product roadmap

## Phase 1 (current)

Phase 1 adds RAG, stateful patient memory, an optional LLM symptom router, intake prompt improvements, and a cardiology specialist.

- **RAG**: Static triage/clinical guideline texts in `backend/knowledge/`; keyword-based retrieval in `backend/services/knowledge.py`. Retrieved context is passed into rule-based triage and into the doctor report (e.g. "Relevant guidelines (excerpt)").
- **Stateful memory**: Auth0 `user.sub` is used as `patient_id`. Per-patient visit summaries are stored in-memory (`backend/services/patient_memory.py`). Previous visit context is injected into the general physician prompt and into specialist first-message context when available.
- **Optional LLM router**: A one-off Backboard call (router prompt) returns a single specialist name (dermatology | dental | cardiology | gastroenterology | general). Rule-based routing in `backend/services/routing.py` is used when the LLM reply is invalid or the call is skipped. Routing includes cardiology and gastroenterology keywords.
- **Intake prompt tweaks**: General physician prompt asks for at least 2–3 follow-ups before considering intake complete; suggests symptom image upload for visible symptoms (rash, swelling, wound); suggests vitals video for breathing/heart/stress-related symptoms.
- **Cardiology specialist**: Cardiology role and Backboard assistant; `backend/prompts/cardiologist_prompt.txt`; specialist handoff flow in main supports dermatology, dental, and cardiology.
- **When intake stops**: Intake stops when all required fields (primary_symptom, location, severity, duration, additional_symptoms) are filled. The backend decides this via `is_intake_complete(thread_id)` in `backend/services/intake.py`; the model is not asked to "decide" when to stop. Extraction runs on every user message (including the first) and can fill multiple fields per message.

## Phase 2 (future)

Deferred work to be tackled after Phase 1:

- **Longitudinal monitoring**: Store prior symptom images and assessments per patient; compare "Day 1 vs Day 3" and support escalation or progression prompts.
- **Full LLM triage agent**: Optional replacement or complement to rule-based triage (e.g. LLM suggests urgency/department, validated or overridden by rules).
- **Full LLM report agent**: Narrative report generated by an LLM instead of (or in addition to) the structured template.
- **Additional specialists**: Gastroenterologist as a formal specialist agent (routing keywords already in place); general practice as a formal specialist agent.
- **Persistent storage for patient memory**: Replace in-memory store with a database.
- **Backboard features**: Explore tool calling for RAG, agent-level documents, and other Backboard capabilities once Phase 1 RAG is in place.
